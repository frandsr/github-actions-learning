Glosario — GitHub Actions

Las 6 piezas del modelo. Definiciones oficiales de GitHub. Documento de referencia: volvé acá cuando una palabra no te cierre.

Workflow

Un proceso automatizado y configurable que corre uno o más jobs. Se define en un archivo YAML dentro de .github/workflows/ de tu repo.

"A configurable automated process that will run one or more jobs."

Event (evento)

Una actividad específica en el repo que dispara la ejecución de un workflow. Ej: un push, abrir un pull_request, un horario (schedule) o un disparo manual (workflow_dispatch).

"A specific activity in a repository that triggers a workflow run."

Job

Un conjunto de steps que se ejecutan en el mismo runner. Los jobs corren en paralelo por defecto; podés hacerlos secuenciales con needs.

"A set of steps in a workflow that is executed on the same runner."

Runner

El servidor (máquina) que ejecuta tus workflows cuando se disparan. GitHub ofrece runners hospedados (Ubuntu, Windows, macOS); también podés tener self-hosted.

"A server that runs your workflows when they're triggered."

Step (paso)

Una tarea individual dentro de un job. Un step o bien corre un script de shell (run:) o bien usa una action (uses:). Los steps comparten el sistema de archivos del runner.

"...either a shell script that will be executed, or an action that will be run."

Action

Una extensión reutilizable que simplifica tu workflow: un bloque empaquetado de lógica que invocás con uses:. Ej: actions/checkout (clona tu repo) o actions/setup-node (instala Node).

"A reusable extension that can simplify your workflow."

La jerarquía, de afuera hacia adentro

Event dispara → Workflow contiene → Jobs corren en → Runners, y cada job tiene → Steps que usan → Actions o corren scripts.

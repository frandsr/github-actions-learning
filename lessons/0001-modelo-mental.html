Lección 1 · Modelo mental

Qué pasa realmente cuando GitHub Actions "corre algo"

Antes de escribir una sola línea de YAML, necesitás el modelo mental. Si entendés cómo encajan las 5 piezas, el YAML después es solo rellenar el formulario.

🏆 Tu victoria de hoy: vas a poder mirar cualquier workflow y decir en voz alta "esto se dispara por tal evento, corre estos jobs en tal runner, y cada job hace estos steps" — sin tocar la sintaxis todavía.

El problema que resuelve (el "por qué")

Vos ya hacés cosas a mano cada vez que tocás el código: correr los tests, pasar el linter, buildear, a veces publicar. Eso es trabajo repetitivo, fácil de olvidar y fácil de hacer mal. CI/CD es la idea de que una máquina haga esas tareas automáticamente cada vez que pasa algo en tu repo.

GitHub Actions es el sistema de GitHub para eso: vos describís "cuándo" y "qué", y GitHub se encarga de levantar una máquina y ejecutarlo. Nada más. Todo el vocabulario que sigue son nombres para las partes de esa frase.

Las 5 piezas (y cómo se enganchan)

Event→ dispara → Workflow→ contiene → Jobs→ corren en → Runner ; cada job = Steps→ usan → Actions

1. Event — el disparador

Algo pasa en tu repo: un push, alguien abre un pull_request, llega un horario programado (schedule), o apretás un botón para correrlo a mano (workflow_dispatch). GitHub lo detecta y se pregunta: "¿hay algún workflow que reaccione a esto?".

Oficial: "A specific activity in a repository that triggers a workflow run." — Understanding GitHub Actions

2. Workflow — la receta

Un archivo YAML que vive en .github/workflows/ de tu repo. Dice: "cuando pase este evento, hacé estos jobs". Un repo puede tener muchos workflows (uno para tests, otro para deploy, otro para un cron nocturno).

Oficial: "A configurable automated process that will run one or more jobs." — About workflows

3. Jobs — las tareas grandes

Un workflow se divide en jobs. Cada job es una unidad de trabajo (ej: "test", "build", "deploy"). Por defecto los jobs corren en paralelo y cada uno arranca en una máquina limpia y aislada. Si querés que uno espere a otro, se lo decís explícitamente.

Oficial: "A set of steps in a workflow that is executed on the same runner." — Understanding GitHub Actions

4. Runner — la máquina

El servidor donde corre un job. GitHub te presta máquinas con Ubuntu, Windows o macOS ya instaladas. Cada job pide su runner (ej: runs-on: ubuntu-latest). Cuando el job termina, la máquina se descarta — no hay estado que sobreviva entre runs.

Oficial: "A server that runs your workflows when they're triggered." — Understanding GitHub Actions

5. Steps + Actions — los pasos concretos

Dentro de un job, los steps se ejecutan en orden, uno tras otro, compartiendo la misma máquina. Cada step hace una de dos cosas:

Tipo de stepCómo se escribeEjemplo
Corre un comando de shellrun:run: npm test
Usa una Action reutilizableuses:uses: actions/checkout@v4

Una Action es lógica empaquetada que alguien ya escribió para que vos no la repitas: clonar tu repo, instalar Node, cachear dependencias. Es la pieza "reutilizable" del sistema.

Oficial: "A reusable extension that can simplify your workflow." — Understanding GitHub Actions · Glosario completo: ver acá

Cómo se ve junto (mirá, no memorices)

No te pido escribir esto todavía. Solo leelo y reconocé las 5 piezas:

name: CI                      # nombre del workflow
on: [push, pull_request]      # 1) EVENT: cuándo se dispara

jobs:
  test:                       # 3) JOB llamado "test"
    runs-on: ubuntu-latest    # 4) RUNNER: máquina Ubuntu
    steps:                    # 5) STEPS, en orden:
      - uses: actions/checkout@v4   # ACTION: clona tu repo
      - run: npm ci                 # comando shell: instala deps
      - run: npm test               # comando shell: corre tests

Todo el archivo es el workflow. ¿Lo leíste y ubicaste cada pieza? Entonces ya cumpliste la victoria de hoy. Probemos con el quiz.

Quiz — 3 preguntas, feedback al instante

1. Abrís un Pull Request y se dispara un workflow. ¿Qué pieza del modelo representa "abrir el PR"?
2. Un job tiene tres steps: checkout, instalar deps, correr tests. ¿Qué es verdad sobre cómo corren?
3. En un step ves uses: actions/setup-node@v4. ¿Qué te dice ese uses:?
💬 Tu tutor sos vos + el agente. ¿Algo no te cerró — por qué los jobs corren aislados, qué es @v4, cuándo conviene push vs pull_request? Preguntámelo en el chat. Para eso estoy.
